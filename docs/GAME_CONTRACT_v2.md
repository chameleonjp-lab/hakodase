# HAKODASE v2 ゲーム契約

## 位置づけ
この文書はv2のゲーム仕様の正本である。現行実装と旧MVP文書が異なる場合、後続実装はこの文書を基準にする。ただし今回は実装しない。

## 一文説明
箱を滑らせ、順番と倉庫設備を読み、同じ印の搬出口からすべて出荷する短時間パズル。

## 残す現行設計
- 箱を滑らせ、同じ色と記号の搬出口から盤外へ出す。
- 箱は壁、他の箱、盤端などで止まるまで一気に滑る。
- 全箱を搬出するとクリアする。
- Canvas2D版を正式な基本版として残す。
- ゲームロジックと描画を分離する。
- `src/core/` はDOMとCanvasへ依存させない。
- Rendererは状態を受け取って描画し、ゲーム状態を書き換えない。
- Pointer Eventsでタッチとマウスを扱う。
- seed付き生成と、同じseedから同じ盤面を再現する方針を残す。
- 生成処理には上限を持たせ、無限ループを禁止する。
- 色だけに頼らず、箱と出口の両方へ同じ記号を出す。
- HTML、CSS、JavaScriptを分けたES Modules構成を正式な例外として残す。
- 外部画像や外部音声を使わずに成立する軽量な描画方針を残す。
- 将来Three.js/WebGLを追加できる分離設計は残す。ただし一行差し替えだけで完全移行できるとは扱わない。

## 用語
- 箱: プレイヤーが滑らせる出荷対象。
- 搬出口: 箱を盤外へ出すための色と記号を持つ出口。出荷シャッターとは別物。
- `swipeCount`: 有効なスライド操作の回数。画面では原則「操作」と表示する。
- `distanceCells`: 箱が通過したマス数の合計。統計や調整用で、公式順位の主条件には使わない。
- `optimalSwipes`: 最短でクリアするために必要なスライド操作数。公式問題の難しさ判定に使う。
- `undoCount`: 戻すを使った回数。

## 20操作契約
旧MVPの「手数」は通過マス数を数えていた。v2では操作回数と移動距離を分離する。

公式問題の「最低20手」は次を意味する。

```text
optimalSwipes >= 20
```

これは「移動距離20マス以上」ではない。公式問題の暫定目標範囲は次とする。

```text
20 <= optimalSwipes <= 35
```

この数値を変更する場合は、決定記録と問題版を更新する。

## 盤面方針
```text
主盤面: 7×9
箱の総数: 暫定8〜14個
色数: 3〜6色
同じ色の箱: 複数配置を許可
搬出口: 1色につき1個を基本とする
```

公式競技では、実行時に未検証の完全ランダム盤面を直接ランキング対象にしない。公式問題は、開発時に生成・検証した問題集から選ぶ。エンドレスではseed付きランダム生成を使える。

## 公式問題の採用条件
- 正確な解法がある。
- `optimalSwipes` が20〜35。
- 最初から全箱を直接搬出できるだけの盤面ではない。
- 正しい順番を考える場面が複数ある。
- 同じ操作の反復だけで終わらない。
- 解法と無関係な壁を大量に置かない。
- 1回の誤操作で回復不能になる盤面を増やしすぎない。
- 間違えた理由を画面上で理解できる。
- 公式問題は `puzzleId`、`rulesVersion`、`generatorVersion`、`boardHash` を持つ。

具体的な版番号と生成方法はPhase 3で実装する。

## モード
### 練習
- 固定された短い問題。
- 基本操作、出荷レーン、出荷シャッターを順番に説明する。
- 戻す操作を使える。
- Supabaseへスコアを送らない。
- 公式ランキングへ記録しない。

### 本日の出荷
- 公式競技モード。
- 全プレイヤーが同じ `puzzleId` の検証済み問題を遊ぶ。
- 3・2・1の後、GOと同時に盤面を表示して計時を始める。
- クリアタイムを競う。
- 初回記録とベスト記録を扱う。
- Supabase連携対象。
- 同じ問題であることを確認できない記録を同じ順位へ混ぜない。

### エンドレス
- seed付きランダム生成。
- seed指定、再挑戦、問題共有を可能にする。
- 公式総合ランキングへ混ぜない。
- 同じseedの端末内記録を扱うことはできる。
- Supabase送信は初回公開では行わない。

## 画面状態
基本状態は `home`、`nameConfirm`、`countdown`、`playing`、`result`。補助状態は `rules`、`ranking`。

### home
タイトル、一文説明、プレイヤー名、本日の出荷、エンドレス、練習、スタート、ルール説明、ゲームをシェア、カメレオンJPの実験場へ戻る導線を持つ。

### nameConfirm
名前の確認または入力を行う。前後空白は保存前に除去する。空文字は未入力として扱い、開始できないか端末内既存名へ戻す。長すぎる名前の最大文字数はTBD。保存場所は端末内。ゲーム開始前にキーボードを閉じる。

### countdown
3、2、1、GOを表示する。公式競技ではカウントダウン中に盤面を見せない。GO、盤面表示、計時開始を同じ開始処理で行い、二重開始を防ぐ。

### playing
常時表示を優先するものは、タイム、操作数、残り箱数、戻す、リタイア、盤面。seed入力、難易度選択、ランキング一覧、長い説明は主画面へ常駐させない。

### result
クリアタイム、操作数、最短操作数との差、初回記録、ベスト記録、ランキング送信状態、同じ問題を再挑戦、次の問題またはモードに応じた次の行動、結果をシェア、ホームへ、カメレオンJPの実験場へ戻る導線を表示する。

## タイマー
v2では競技タイマーの基準に `Date.now()` を使わない。ブラウザの経過時間は `performance.now()` で測る。

### 公式競技
- GOと同時に開始する。
- 盤面を見てから最初の操作をするまでの時間も含める。
- クリア判定が確定した瞬間に止める。
- クリア演出時間は含めない。
- タブ移動、画面ロック、別アプリへの移動などでページが隠れた場合、その試行を無効にする方針を基本とする。
- 無効化時には理由と再挑戦方法を表示する。

### 練習・エンドレス
ページが隠れた場合の一時停止を許可できる。ただし公式競技と同じ記録へ混ぜない。

経過時間と描画フレーム数は分離する。

## 入力
- 箱へ触れた時、視覚反応をすぐ返す。
- Pointer Eventsを使う。
- `pointerup` で操作を確定する。
- `pointercancel` と `lostpointercapture` では操作を破棄し、箱を元の表示位置へ戻す。
- 画面回転、アプリ切替、2本目の指などで意図しない移動を確定しない。
- 移動アニメ中の連続入力を制御する。
- 初回実装では全入力を短くロックする安全な方式を採用し、将来必要なら予約入力を検討する。
- 無効方向には小さな揺れなどで理由を伝える。
- 論理座標は整数グリッド。
- アニメ終了時に正しいセルまたは出口位置へ正確に合わせる。
- ピンチズーム、ダブルタップ拡大、長押しコピー、横スクロールなど、プレイを壊すスマホ誤操作を抑える。
- 通常のリンクには、必要な長押し操作を許可する。

## 戻す・リタイア・詰み
### 戻す
1操作戻せる。タイマーは戻さない。箱位置、操作数、距離、シャッター状態を正確に戻す。`undoCount` を持つ。公式競技でも使用できる。公式順位はクリアタイムを主とし、初回実装では戻す専用の時間加算を設けない。

### リタイア
プレイ中に明示的に終了できる。リタイア記録をクリアタイムランキングへ送らない。結果またはホームへ安全に戻れる。通信やアニメーションを二重に残さない。

### 詰み
少なくとも「合法な操作が一つもない」状態を検出する。「出荷できなくなりました」と理由を示す。戻す、やりなおす、リタイアを選べる。複雑な完全詰み判定は、公式問題の事前検証で減らす。

## 独自ギミック
### 出荷レーン
既存の一方通行床に相当する。矢印方向へ滑っている箱だけ通れる。逆方向や横方向からの進入は通さない。色に頼らず大きな矢印で分かる。1問へ過剰に置かない。

### 出荷シャッター
可動ギミック。実時間ではなく、有効なスライド操作数で開閉する。判定には `distanceCells` ではなく `swipeCount` を使う。開閉周期と位相をデータとして持つ。次の操作で開くか閉じるかを画面で確認できる。待つだけで攻略できる実時間式は採用しない。出口の「搬出口」と混同しない名称と見た目にする。

## 後続Phase
Phase 1で現行コードの正しさをv2へ合わせる。Phase 2以降で画面遷移、生成、Supabase、独自ギミック、公開判定を別目的として扱う。

## RendererとframeState契約
- Rendererは `init`、`resize`、`render`、`destroy`、`clientToCell` 相当の責務を持つ。
- `frameState` は描画用の読み取りスナップショットであり、Rendererはゲーム状態を書き換えない。
- `frameState` には盤面、箱位置または表示位置、選択状態、プレビュー、演出状態、タイム表示に必要な値を含められる。
- Canvas2D版を正式なフォールバックとして残す。
- Three.js版でもゲームロジック、生成、ランキングを再利用できる構造を目指す。
- 3D化には、カメラ、光源、立体物管理、ヒット判定、画面サイズ変更、高解像度負荷、資源破棄、WebGLコンテキスト消失と復帰が必要になる。
- Three.js化はRenderer生成箇所だけを一行変えて完成ではない。
- WebGLが使用できない場合、Canvas2Dへ戻す設計を持つ。
- 必要な機能が使えるかを実行時に確認する。
- ブラウザ名や端末名だけで処理を分けない。
- Three.js導入は、競技ルール、画面遷移、盤面生成、ランキングが完成した後の別作業とする。

## ファイル構成と版管理
HAKODASEは、開発用ソースを複数ファイルで維持する。

```text
index.html
styles/
src/
test/
docs/
scripts/
```

これは、保守、単体テスト、盤面生成、将来の描画差し替えのための正式な例外である。Codeberg Pages公開用に1ファイル成果物が必要と判断された場合は、後続作業で自動生成する方法を検討する。開発用分割コードと公開用1ファイルを手作業で二重管理してはいけない。

- import対象のソースファイル名は安定させる。
- 古いソースを残すためだけに `engine_v2.js` などを増殖させない。
- 新しい契約文書には `_v2.md` を付ける。
- 次の大きな契約変更では `_v3.md` へ進める。
- リリース版はタグ、ZIP名、`CLIENT_VERSION`、`rulesVersion`、`generatorVersion`で識別する。
