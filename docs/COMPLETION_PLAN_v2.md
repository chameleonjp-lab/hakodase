## 2. 「完成」の定義

HAKODASE v2を完成と呼べるのは、次をすべて満たした時だけである。

1. `home → nameConfirm → countdown → playing → result` が競合なく動く。
2. 練習、本日の出荷、エンドレスが分離され、公式記録と非公式記録が混ざらない。
3. 本日の出荷は、同じ `puzzleId` の検証済み問題を全プレイヤーへ出す。
4. 公式問題は厳密な最短解を持ち、`20 <= optimalSwipes <= 35` を満たす。
5. 同色の複数箱、出荷レーン、出荷シャッターを含むHAKODASE独自の判断が成立する。
6. GOと盤面表示と計時開始が同じ処理で行われる。
7. 戻す、リタイア、詰み、中断、再挑戦、共有、ランキング失敗を安全に処理できる。
8. Supabaseへ公式クリア時だけ1プレイ1回送信し、初回記録とベスト記録を表示できる。
9. 異なる問題ID、ルール版、生成版の記録が同じ公式順位へ混ざらない。
10. iPhone SE級の320px幅、iPhone 11 Pro、iPhone 17 Pro、iPad Pro 2018を含む確認対象で、画面と操作と性能の合格記録がある。
11. Codeberg Pagesの公開URL、ゲーム内URL、シェアURL、Supabase登録、実験場トップ、詳細ランキングが一致する。
12. 公開候補版の自動テスト、実機テスト、性能試験、セキュリティ確認がすべて合格する。
13. リリースタグ、公開物、テスト報告、ロールバック先を特定できる。

一つでも未確認なら「完成」ではなく「公開候補」または「未確認」と記録する。

## 3. 変更管理の厳格ルール

### 3.1 Pull Request単位

- 1つのPull Requestで達成する目的を1つに絞る。
- 各Pull Requestは、直前の統合済み基準ブランチから作る。
- 基準ブランチへ直接コミットしない。
- Draft Pull Requestとして作り、差分確認と合格条件確認の前にマージしない。
- 目的外のリファクタ、依存追加、画面変更、ランキング変更を同梱しない。
- 仕様変更が必要なら、コードより先に正本文書と`DECISION_LOG.md`を更新する。
- `CURRENT_TASK.md`は現在作業だけを示し、前Phaseを現在作業のように残さない。

### 3.2 各Pull Requestの必須報告

Pull Request本文へ、次を必ず書く。

1. 今回の一目的
2. 基準ブランチと基準コミット
3. 変更したファイル
4. 変更しなかった重要範囲
5. 実装内容
6. 実行した検証と件数
7. ブラウザ確認の有無
8. 実機確認の有無
9. 既知の問題
10. 次回へ引き継ぐ内容
11. ロールバック方法

### 3.3 統合禁止条件

次のいずれかがあるPull Requestはマージしない。

- `npm test`失敗
- `git diff --check`失敗
- 目的外差分
- 仕様と実装の不一致
- 公式問題の厳密解未確認
- Supabaseの実仕様を推測した変更
- secret key、service role key、管理用トークンの混入
- ブラウザ未確認なのに画面完成を主張
- 実機未確認なのに対応端末を保証
- 異なる問題を同じランキングへ送る可能性
- クリア以外の送信、二重送信、古い通信結果の混入
- 100ms超の意図しない停止を既知のまま放置

## 5. Phase 2：画面遷移と3モード

### 5.1 P2-01 アプリ状態機械

**目的**: DOM操作、ゲーム進行、通信、演出を一つの巨大クラスへ混ぜず、状態遷移を一元管理する。

推奨責務:

```text
AppController
画面状態と画面遷移。

RunController
1プレイの開始、無効化、クリア、リタイア、結果確定。

GameEngine
盤面ルール、位置、操作数、距離、undo。

AnimationController
表示位置、入力ロック、退場・クリア演出。

RankingService
記録の保存・取得。Phase 2ではローカルのみ。
```

必須状態:

```text
home
nameConfirm
countdown
playing
result
rules
ranking
```

必須防御:

- 画面遷移ごとに`playId`または同等の世代識別子を持つ。
- 古いタイマー、古いPromise、古いイベント結果は現在の世代へ作用させない。
- クリア、リタイア、タブ非表示、再挑戦が同時に起きても結果遷移は一度だけ。
- `destroy()`または画面離脱処理でイベントとタイマーを解除する。

**完了条件**:

- 許可された遷移と禁止遷移を単体テストできる。
- `playing`以外で箱を動かせない。
- `result`表示後に再度結果処理が走らない。
- 旧MVPの暫定開始処理を、Phase 2の正式開始へ置き換えられる。

### 5.2 P2-02 ホーム・名前・モード

ホームへ置くもの:

- タイトルと一文説明
- プレイヤー名
- 本日の出荷
- エンドレス
- 練習
- ルール
- ゲームをシェア
- 実験場へ戻る

名前処理:

- 前後空白を除く。
- 空文字では開始しない。
- 最大文字数は、共通ランキング仕様を確認して決定する。確認前に推測しない。
- 端末内へ保存する。
- カウントダウン前に入力欄のfocusを外し、ソフトウェアキーボードを閉じる。
- 表示用文字列は`textContent`で扱い、HTMLとして解釈しない。

**完了条件**:

- 3モードの選択がRun設定へ正しく渡る。
- 名前未入力、長すぎる名前、空白だけ、絵文字を含む名前を試験する。
- 名前確定後に320px幅で画面が横へ動かない。

### 5.3 P2-03 カウントダウン・公式時計

公式モード:

1. 盤面を隠した状態で3・2・1を表示する。
2. GOと盤面表示を同じ開始処理で行う。
3. 同じ処理内で`engine.start(performance.now())`を1回だけ呼ぶ。
4. クリア判定時の単調時刻で終了する。
5. クリア演出は記録へ含めない。
6. `visibilitychange`でページが隠れた公式試行を無効にする。

練習・エンドレスは一時停止を許可できるが、公式と同じ記録へ混ぜない。

**完了条件**:

- カウントダウン前と途中は経過0。
- GO後だけ増える。
- 二重タップでも開始は1回。
- 背景移動した公式試行は送信対象にならない。
- やりなおしで古いカウントダウンが後から発火しない。

### 5.4 P2-04 プレイ画面

常時表示:

```text
タイム
操作数
残り箱数
戻す
リタイア
盤面
```

プレイ中に常駐させないもの:

```text
seed入力
難易度選択
ランキング一覧
長い説明
```

undo:

- GameEngineの履歴APIをUIへ接続する。
- タイマーは戻さない。
- undo後の表示位置と論理位置を同期する。
- undo中の連打を防ぐ。
- 履歴なし、クリア後、入力ロック中の扱いを表示する。

リタイア:

- 公式スコアを送らない。
- クリア確定後は受け付けない。
- 古いクリア演出や結果タイマーを解除する。
- 誤タップを避ける確認方法は、実機で操作負担を見て決める。

詰み:

- 少なくとも合法操作0件を検出する。
- 戻す、やりなおす、リタイアを提示する。
- 完全詰みの重い実行時探索は必須にせず、公式問題の事前検証を優先する。

### 5.5 P2-05 結果・共有

結果表示:

- クリアタイム
- `swipeCount`
- `optimalSwipes`との差
- `distanceCells`
- `undoCount`
- モードと問題識別情報
- ローカル初回・ベスト
- 通信状態の置き場所（Phase 4で接続）

共有:

1. Web Share対応時は共有を試す。
2. 利用者キャンセルはエラー表示しない。
3. 共有失敗時はClipboardへ移る。
4. Clipboard失敗時は選択可能なテキスト表示へ移る。
5. すべて失敗しても結果画面を使える。

共有文へ含めるもの:

- ゲーム名
- 記録
- 操作数
- 公式なら問題IDまたは日付
- 正式公開URL

公開URL未確定の間は仮URLを本番文として固定しない。

### 5.6 Phase 2 Gate

Phase 3へ進む条件:

- 7状態の遷移テストが通る。
- 320×568、390×844、PCで主要フローを実ブラウザ確認済み。
- コンソールエラー0。
- START前にタイマーが進まない。
- 公式のタブ非表示は無効になる。
- undo、リタイア、再挑戦、共有の競合がない。
- プレイ中に不要な管理UIが出ていない。
- Phase 1のブラウザ未確認項目を消化している。

## 6. Phase 3：盤面生成第2世代

### 6.1 P3-01 データ契約

盤面へ最低限持たせるもの:

```text
schemaVersion
rulesVersion
generatorVersion
puzzleId
boardHash
width / height
blocks
walls
gates
lanes（Phase 5まで空を許可）
shutters（Phase 5まで空を許可）
expectedOptimalSwipes
```

`boardHash`は、配列順や空白差で変わらない正規化済みデータから作る。同一内容なら同一、内容が違えば実用上区別できる値にする。

**破壊的変更規則**:

- ルール解釈が変わる変更では`rulesVersion`を上げる。
- 生成規則が変わる変更では`generatorVersion`を上げる。
- 盤面内容が変わったら同じ`puzzleId`を使い回さない。
- 既にランキング記録がある問題の内容を上書きしない。

### 6.2 P3-02 ソルバー

必須能力:

- 1スライド=1コストの厳密最短操作。
- 解法列を返す。
- 同じ盤面では同じ解法列を返す。
- ノード上限、時間計測、終了理由を返す。
- 候補検査用であり、プレイの毎フレームでは実行しない。

性能が足りない場合の順序:

1. 状態表現を圧縮する。
2. 対称状態と重複を除く。
3. 明らかな下界で枝を減らす。
4. 公式問題生成を完全にオフラインへ寄せる。
5. それでも不足する場合、箱数や候補規則を縮小し、契約変更として記録する。

正しさを確認せず、推定値を`optimalSwipes`として保存してはいけない。

### 6.3 P3-03 生成器

暫定範囲:

```text
主盤面: 7×9
箱: 8〜14個
色: 3〜6色
同色の複数箱: 許可
公式候補: 20〜35最短操作
```

品質指標:

- 最短操作数
- 解法数または序盤分岐数
- 初手候補数
- 詰みへ入る操作数
- 解法に使われない壁の割合
- 同じ操作の反復率
- 盤面占有率
- 出口まで直行できる箱の初期数
- 解法の色・方向の偏り

無限ループを禁止し、試行上限と理由付き失敗を返す。フォールバックは検証済み固定問題を使用する。

### 6.4 P3-04 大量検査

開発時に1000件以上の候補を検査する。

検査出力:

```text
seed
boardHash
optimalSwipes
solver nodes
solver duration
初手数
詰み指標
壁利用率
採用/不採用
不採用理由
```

同じ`boardHash`は重複として除外する。大量検査は公開ブラウザで実行せず、`scripts/`の開発処理にする。

### 6.5 P3-05 公式問題集

暫定目標:

- 自動条件を満たす候補を十分に作る。
- 人による試遊を経た問題を30問以上用意することを目標にする。
- 初回公開に必要な問題数は、試遊結果と日替わり運用方法を確認してDecision Logで確定する。

各問題へ残す証拠:

- 厳密最短操作数
- 代表解法
- 生成版とルール版
- boardHash
- 手動試遊者と評価
- 既知の詰み方
- 採用理由

### 6.6 P3-06 本日の出荷

- 日本時間の日付から決定論的に問題を選ぶ。
- 端末の時計だけを無条件に信用しない方法はPhase 4のサーバー構成と合わせて決める。
- サーバー時刻を使えない初回版では、固定公式問題へ限定する安全案を残す。
- 日付境界、再読み込み、オフライン、時刻ずれをテストする。
- エンドレスはseed共有を許可するが公式総合順位へ送らない。

### 6.7 Phase 3 Gate

- 1000件以上の検査報告がある。
- 公式問題候補は全件厳密解あり。
- 全件20〜35操作。
- boardHash重複なし。
- 同じ問題IDから同じ盤面が出る。
- プレイ開始時に重いソルバーを実行しない。
- 公式問題集の手動試遊結果がある。
- ルール版変更時に旧ランキングへ混ざらない。

## 7. Phase 4：Supabaseランキング

### 7.1 P4-01 現状監査

実装前に実物を確認する。

- `public.games`
- スコア表
- `submit_score`
- `get_first_try_ranking`
- `get_best_score_ranking`
- `get_game_play_stats`
- 引数名と型
- 返却列
- `rank_no`の同率順位
- 行単位のアクセス制御
- Publishable keyで許可される範囲
- `puzzleId`または`challenge_id`対応

確認できないものは未確認とし、推測でSQL、RPC、列を作らない。

### 7.2 P4-02 問題IDの分離

判断順:

1. 現行RPCで問題IDを安全に分離できるか確認する。
2. できる場合は現行基盤へ合わせる。
3. できない場合は、他ゲームへ影響しないSQL案を別途提示する。
4. ユーザー承認前に本番DBへ適用しない。
5. DB変更なしで公開する場合は、公式問題を固定1問に限定し、同一条件を保つ。

### 7.3 P4-03 通信契約

送信対象:

```text
本日の出荷の有効なクリアだけ
```

送信しないもの:

```text
練習
エンドレス
リタイア
タブ非表示で無効化された試行
未検証問題
```

必須:

- `p_score`は1/100秒の整数。
- 結果確定後に1回だけ送る。
- `playId`で古い応答を無視する。
- timeoutを設定する。
- 結果画面を先に表示する。
- 送信中、成功、失敗、空、未設定を区別する。
- 再送が必要な場合も重複登録条件を確認してから行う。
- secret key、service role keyをブラウザへ入れない。

### 7.4 P4-04 結果表示

- 初回記録
- ベスト記録
- 現在順位
- プレイ回数と参加人数
- 同じ問題IDのランキング
- 詳細ランキング導線

単位・並び:

```text
score_order: asc
score_scale: 100
score_decimals: 2
```

### 7.5 Phase 4 Gate

- 実Supabaseで送信、初回、ベスト、順位、回数を確認済み。
- 1クリア1件だけ保存される。
- 異なる問題IDが混ざらない。
- 通信失敗でも結果と再挑戦が使える。
- 異常値、空名、長すぎる名をサーバー側でも拒否または正規化する。
- 公開クライアントにPublishable key以外の秘密情報がない。
- RLSとRPC権限の確認記録がある。

## 8. Phase 5：独自ギミック

### 8.1 出荷レーン

- 矢印方向の移動だけ通す。
- 逆方向と横方向は停止する。
- 出発、進入、通過の各場合をルールとして固定する。
- 色ではなく形と矢印で伝える。
- 箱選択時のプレビューも同じ判定を使う。
- Engine、Solver、Rendererが同じ純粋ルールを使う。

### 8.2 出荷シャッター

状態例:

```text
period
openFor
phase
現在のswipeCount
```

- 実時間ではなく`swipeCount`で状態を求める。
- 無効操作では状態を進めない。
- undoで箱位置、操作数、シャッター状態を同時に戻す。
- 次の操作後の状態を見えるようにする。
- 待つだけの攻略を作らない。
- ソルバー状態へ必要な操作数剰余を含める。

### 8.3 問題集への反映

ギミック追加はルール版変更として扱う。既存の公式問題IDとランキングを再利用しない。

最低限の手動試遊:

- レーンのみ
- シャッターのみ
- 両方
- undoを使うケース
- 誤操作から回復するケース
- 詰みになるケース

### 8.4 Phase 5 Gate

- 表示を見て進行方向と開閉状態を判断できる。
- EngineとSolverの判定が一致する。
- undo後の状態が完全一致する。
- ギミック問題にも厳密最短解がある。
- 1問へ要素を詰め込みすぎていない。
- チュートリアルで基本から順に学べる。

## 9. Phase 6：品質保証

### 9.1 自動ブラウザ試験

最低対象幅:

```text
320×568
375×667
390×844
768×1024
PC
```

主要シナリオ:

- 名前入力
- 3モード選択
- カウントダウン
- START前0ms
- 公式クリア
- undo
- リタイア
- 詰み
- タブ非表示
- 画面回転相当
- pointercancel
- 2本指
- 再挑戦
- 共有成功、キャンセル、失敗
- ランキング成功、空、失敗、timeout、未設定
- クリア直後の連打
- 旧通信が次プレイへ戻るケース

新しいE2E依存を追加する場合は、開発専用であること、公開物へ含まれないこと、導入理由をDecision Logへ記録する。

### 9.2 性能・継続試験

測定する最重条件:

```text
7×9
箱14個
色6色
壁最大候補
レーン最大候補
シャッター最大候補
粒子最大候補
ランキング状態表示中
```

合格目標:

- 60Hz端末で滑らかな60fpsを目標。
- 高更新端末でもアニメ時間が変わらない。
- 入力後50ms以内の視覚反応を目標。
- 50ms超の処理は原因を記録し調査する。
- 100ms超の意図しない停止は公開停止条件。
- 連続再挑戦10回でイベント、配列、粒子、描画資源が増え続けない。
- 30分連続プレイで劣化しない。
- バックグラウンド移動と復帰10回で壊れない。
- 通常より20%重い条件を3回連続で合格する。

見た目を減らす場合は、粒子、影、ぼかしから調整し、操作、判定、問題の公平性を削らない。

### 9.3 実機試験

必須候補:

- iPhone SE級の320px幅
- iPhone 11 Pro
- iPhone 17 Pro
- iPad Pro 2018縦
- iPad Pro 2018横
- Safari
- PC主要モダンブラウザ

確認:

- 横スクロール0
- 盤面と主要ボタンが見切れない
- Safari下部バー変化
- safe area
- キーボード閉鎖
- 長押し、ダブルタップ、ピンチ、2本指
- 画面回転
- 低速回線
- 一時オフライン
- Web Share対応/非対応
- 30分継続

実機で確認していない端末は保証済みと書かない。

### 9.4 アクセシビリティ

- 箱と出口を色と記号で対応させる。
- フォーカス位置を画面遷移時に適切に移す。
- ボタンは44 CSS px程度の押しやすい領域を目標にする。
- 状態メッセージへ`aria-live`を使う。
- `prefers-reduced-motion`で粒子、揺れ、大きな移動を弱める。
- ページ全体の拡大を必要以上に禁止しない。盤面のジェスチャー抑制と通常リンクを分ける。
- 文字サイズとコントラストを実画面で確認する。

## 13. リスク表

| リスク | 兆候 | 予防 | 発生時の判断 |
| --- | --- | --- | --- |
| ソルバー状態爆発 | 候補1件が上限到達、生成が止まる | オフライン検証、状態圧縮、候補制限 | 推定値で公開せず、箱数・規則を縮小して契約更新 |
| 問題が単調 | 最短操作は長いが同じ動作の反復 | 品質指標と手動試遊 | 自動条件だけで採用しない |
| 日替わり不公平 | 端末ごとに問題が違う | puzzleId、boardHash、版、時刻境界テスト | 固定問題へ戻す |
| 二重送信 | 1クリアで複数行 | playId、送信済みフラグ、RPC側重複防止 | ランキング公開を停止し修正 |
| 古い通信の混入 | 次プレイ結果へ前回順位が表示 | 世代IDで無視 | 結果表示をローカル記録だけへ退避 |
| DB変更が他ゲームへ影響 | 共通RPCの返却や引数が変わる | 監査と別RPC/後方互換 | 本番適用を止め、SQLを再設計 |
| 320pxで操作不能 | ボタンや盤面が隠れる | プレイ中情報を最小化 | 新機能よりレイアウト修正を優先 |
| 高DPRで重い | フレーム落ち、端末発熱 | DPR上限候補、静的層キャッシュ、粒子上限 | 視覚効果を段階的に減らす |
| ギミックとundo不一致 | 戻した後にシャッター状態だけ違う | 状態をswipeCountから純粋計算 | 公開問題からギミックを外し修正 |
| 公開先の版ずれ | GitHub、Codeberg、DBで版が違う | CLIENT_VERSIONとリリース記録 | 公開停止、前安定版へ戻す |
| 秘密情報混入 | service key、管理トークンが差分に出る | 秘密検査、Publishable keyのみ | 直ちに鍵を失効し履歴対応を行う |

## 15. v2.0完成まで入れないもの

次は初回完成を遅らせ、問題の切り分けを難しくするため、v2.0の必須対象にしない。

- Three.js/WebGL版レンダラー
- WebAssembly
- 物理エンジン
- TypeScript全面移行
- PWA
- ログイン
- クラウドセーブ
- リアルタイム対戦
- シーズン制
- 高度な不正防止
- 外部画像・大量音声素材
- ルールと関係しない収集・課金要素

Three.jsは、Canvas2D版がこの計画の最終公開条件を満たした後、別の体験契約、性能契約、フォールバック試験を作って判断する。
