# HAKODASE v2 Git基準・ブランチ運用方針

## 1. この文書の役割

この文書は、HAKODASE v2で使用する正式な統合基準、作業ブランチ、Pull Request、リリース、ロールバックの運用を定める。

ゲーム仕様は各v2契約文書を正本とする。この文書はGit運用だけを扱う。

## 2. 正式な基準ブランチ

正式な統合・リリース基準は次とする。

```text
main
```

2026年7月18日時点で、`main` は次の最新統合コミットへfast-forwardで同期した。

```text
c031eb089b9291880ff2e958e38c3f1e9a79de48
```

このコミットには、Phase 0のv2契約、Phase 1の中核修正、完成までの計画表が含まれる。

以後の実装Pull Requestは、明示的に`main`をbaseとする。

## 3. GitHub既定ブランチとの関係

2026年7月18日の確認時点では、GitHubのリポジトリ既定ブランチ表示が次の旧作業ブランチを指している。

```text
claude/hakodase-puzzle-mvp-82p4qs
```

この旧作業ブランチを、今後の正式な統合基準として扱わない。

GitHubの既定ブランチ設定は、リポジトリ設定で`main`へ変更することを推奨する。ただし、設定変更が完了するまでも、すべてのPull Requestでbaseを明示的に`main`へ指定すれば開発は継続できる。

自動化や開発エージェントは、GitHubの既定ブランチ名だけを信頼してはいけない。作業開始前に`main`の先端と最新統合コミットを確認する。

## 4. 作業ブランチ

各作業は、最新の`main`から新しいブランチを作る。

推奨形式:

```text
agent/<短い目的>
codex/<短い目的>
claude/<短い目的>
fix/<短い目的>
feat/<短い目的>
docs/<短い目的>
```

例:

```text
agent/hakodase-p2-01-app-state
```

一つのブランチへ、複数Phaseの目的を混ぜない。

## 5. Pull Request

- baseは必ず`main`を明示する。
- 原則としてDraft Pull Requestで作成する。
- 1つのPull Requestで達成する目的を1つに絞る。
- テスト、差分、対象外変更、既知の問題を本文へ記録する。
- ユーザーの明示確認前にDraftを解除しない。
- ユーザーの明示確認前にマージしない。
- 自動マージを設定しない。
- `main`へ直接コミットまたは直接pushしない。
- force push、履歴の書き換え、破壊的なrebaseを行わない。

## 6. 推奨するブランチ保護

GitHub設定で、`main`に次の保護を設定することを推奨する。

- Pull Request経由の変更を必須にする。
- マージ前のレビューを必須にする。
- 新しいコミットで古い承認を無効にする。
- 必須テストがある場合は、その成功をマージ条件にする。
- force pushを禁止する。
- ブランチ削除を禁止する。
- 管理者にも同じルールを適用するかを検討する。

この文書を書いた時点では、これらのリポジトリ設定が実際に有効になっているとは断定しない。設定画面での確認が必要である。

## 7. 統合順序

1. 最新の`main`から作業ブランチを作る。
2. 一目的だけを実装する。
3. 必須テストと静的確認を行う。
4. Draft Pull Requestを`main`向けに作る。
5. 差分、テスト、ブラウザ確認、実機確認の状態を確認する。
6. ユーザーが確認してからマージする。
7. マージ後、次の作業は新しい`main`から始める。

前の作業ブランチを次の作業の基準として連鎖させない。

## 8. リリース線

- 正式リリース候補は`main`上のコミットからだけ作る。
- リリースタグは`main`上の検証済みコミットへ付ける。
- Codeberg Pagesへ公開する版は、対応するGitHubコミットを記録する。
- `CLIENT_VERSION`、`rulesVersion`、`generatorVersion`、公式問題集の版をリリース報告と一致させる。
- 作業ブランチ上の未統合コードを正式公開しない。

## 9. ロールバック

- `main`の履歴をforce pushで戻さない。
- 問題のある変更は、原則としてrevert用Pull Requestで取り消す。
- 公開前に、直前の正常なリリースタグまたはコミットを記録する。
- Codeberg Pages、Supabase設定、公式問題集、ゲームコードの戻し先を同じリリース報告へ記録する。

## 10. ブランチ整理

- マージ済み作業ブランチは、必要な確認後に削除できる。
- `main`と同一内容の旧作業ブランチは、既定ブランチ変更と参照確認が終わるまで削除しない。
- 未マージ差分があるブランチを、名前だけで不要と判断しない。
- 削除前に`main`との差分とPull Requestの状態を確認する。
