# P3-01 Decision Record: 盤面データv2

日付: 2026-07-22

## 決定

1. 正式な盤面schemaを`hakodase.board/2`とする。
2. 現行スライド搬出ルールを`slide-exit/1`とする。
3. `schemaVersion`、`rulesVersion`、`generatorVersion`を別々に管理する。
4. 盤面内容の識別には正規化payloadのSHA-256を使い、`boardHash`を`sha256:<hex>`形式で保存する。
5. `boardHash`には盤面挙動と初期状態を含め、`puzzleId`、`generatorVersion`、`expectedOptimalSwipes`は含めない。
6. 箱は一意な文字列`id`を持ち、同色の複数箱を正式に許可する。
7. 使用色ごとに搬出口をちょうど1件必要とする。
8. 公式候補profileは7×9、8〜14箱、3〜6色、同色複数箱、20〜35操作を必須とする。
9. 公式profileの`expectedOptimalSwipes`は値域だけを確認し、厳密解との一致証明はP3-02へ分ける。
10. レーンは現行Engineの`oneway`へ変換する。
11. シャッターはP3-01では保存・構造検査だけを定義し、未対応Engineで黙って無視しない。
12. 同じ`puzzleId`の盤面内容を書き換えず、変更時は新しい`puzzleId`を発行する。

## 判断理由

- 盤面の配列順やJSON空白を同一性判定へ混ぜると、同じ問題が別問題として重複する。
- `puzzleId`だけでは、誤更新や同名別盤面を検出できない。
- generatorVersionだけが変わって同じ盤面が生成された場合、それを別盤面として数えるべきではない。
- 最短値の記録誤りを修正しただけで盤面hashが変わると、同じ盤面のランキングが分断される。
- 同色複数箱は8〜14箱・3〜6色の正式設計に必要であり、配列indexを永続識別子として使えない。
- 将来ギミックを保存できても、Engineが未対応のまま無視するとSolverとプレイ結果が一致しなくなる。

## 代替案として採用しなかったもの

### 32bit seed hashをboardHashとして使う

衝突耐性と改ざん検出が不足するため不採用。

### puzzleIdだけで盤面を識別する

同じIDへの誤上書きを検出できないため不採用。

### Web Cryptoだけを使う

非同期APIであり、生成・検証・Nodeテストの同期境界が複雑になるため不採用。小さい盤面payload向けの同期SHA-256を共通実装する。

### シャッターを空配列専用にする

Phase 5でschema変更が必要になるため不採用。構造は先に定義し、実行未対応時は明示拒否する。

## 影響

- P3-02のソルバーfixtureは盤面データv2を正本として読み込む。
- P3-03の生成器はv2 draftを生成し、materialize時にhashを付ける。
- P3-04の1000件検査は`boardHash`で重複を除外する。
- P3-05の公式問題集は読込時にhashを再計算する。
- Phase 4ランキングでは`puzzleId`と`rulesVersion`を比較条件にし、同じIDでhashが異なる問題を拒否する。

## 未解決

- 8〜14箱で厳密ソルバーが実用時間に収まる状態表現。
- 品質指標の採用閾値。
- 公式問題の必要件数と試遊評価形式。
- シャッターの最終的な時間・手数状態契約。
